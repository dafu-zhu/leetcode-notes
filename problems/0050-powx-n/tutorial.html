Pow(x, n)

Medium leetcode.com ↗
Idea: right approach, wrong execution

Your instinct is correct. Squaring the base to halve the exponent is binary exponentiation, and it is the intended O(log n) solution. Keep the idea.

The mistake is in the odd exponent step. When the exponent is odd, you owe one extra multiply by the current base. Your code instead counted the odd steps in an integer and multiplied by x0 raised to that count. That correction grows as the recursion goes deeper, and it ignores the sign of n, so the result is only correct for some inputs.

That is the whole critique. We keep the idea and rebuild the code from the first read.

Solving it from scratch

Here is the reasoning as it would actually go, starting from the first time I read the problem.

Read itThe task is to compute x raised to the power n. The constraints matter: n can be as large as about two billion, and it can be negative or zero. Multiplying x by itself n times would take up to two billion steps, which is too slow. So the real question is whether I can compute x^n with far fewer multiplications.

Find the key ideaI want to reuse work instead of repeating it. The identity that helps is x^n = (x·x)^(n/2). Squaring the base once lets me halve the exponent. If I keep halving, the number of steps drops from n to about log₂ n, which is roughly 31 steps instead of two billion. That is the core idea: when the exponent is large and I need something faster than linear, I square the base to halve the exponent.

Handle an odd exponentWhat if n is odd? Then n/2 is not exact. I split off one factor: x^n = x · (x·x)^((n-1)/2). So an odd exponent costs one extra multiply by the current base, and after that the exponent is even again. This is the exact step the original code got wrong.

See it as binaryThere is a cleaner way to see the whole thing, and it rests on two facts. First, every exponent is a sum of powers of two, because that is what binary notation means: 13 = 1101₂ = 8 + 4 + 1. Second, x^(a+b) = x^a · x^b, so adding exponents is the same as multiplying powers. Put together, they split the power along the 1 bits of n: 3^13 = 3^(8+4+1) = 3^8 · 3^4 · 3^1.

Build the powers by squaringComputing 3^8 and 3^4 might sound like extra work, but it is not, because each value is the square of the one before it:

3^1 = 3
3^2 = 3^1 × 3^1 = 9
3^4 = 3^2 × 3^2 = 81
3^8 = 3^4 × 3^4 = 6561

Squaring a power doubles its exponent, since (3^a)^2 = 3^(2a). So four squarings give me every value I need. In the loop, a single variable holds the base and squares itself on each pass, taking the values 3, 9, 81, 6561. On each pass I look at one bit of n, reading from the lowest bit with n & 1. When that bit is 1, I multiply the current base into the answer. When it is 0, I skip it.

Sign and zeroFor a negative exponent, x^(-n) = (1/x)^n. I handle this once, at the top, by replacing x with 1/x and n with -n. After that, the loop only ever sees an exponent that is zero or positive. Zero needs no special case: the loop does not run and the answer stays 1.

Write itStart with result = 1 and let x be the running base. While n is not zero: if the lowest bit is 1, multiply the base into result; then square the base and shift n right by one bit. When n reaches zero, result holds the answer.

The algorithm in one sentence When the current bit is 1, multiply the answer by the current base. On every step, square the base and drop one bit. There is no counter and no sign handling inside the loop.

The full solution

class Solution:
    def myPow(self, x: float, n: int) -> float:
        if n < 0:              # x^(-n) == (1/x)^n, so flip the sign once, up front
            x, n = 1 / x, -n
        result = 1.0
        while n:
            if n & 1:          # lowest bit is 1: multiply in the base at this position
                result *= x
            x *= x             # square the base for the next binary position
            n >>= 1            # drop the bit we just handled
        return result

Time is O(log n), one iteration per bit of n. Space is O(1). The loop is iterative, so there is no recursion depth concern even at n = 2^31. This solution ships with a 500 case fuzz suite in test_solution.py.

Takeaway

Pattern to carry forward

When the input is an exponent, or any task that applies the same operation n times with n very large, do not repeat the operation n times. Combine results to cut the count in half each round: square the base and halve the exponent. The answer is the product of the base at each 1 bit of n.

The same structure powers modular exponentiation (pow(a, b, m)), fast matrix power (which computes Fibonacci in O(log n)), and binary lifting (jumping 2^k ancestors in a tree). Learn the shape once and it comes back often.

One lesson from the earlier attempts: if a change makes a wrong solution pass more test cases but you cannot state the rule it now follows, that is not real progress. Write the one sentence rule first. If a piece of code does not serve that rule, remove it.
