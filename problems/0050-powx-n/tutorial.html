Pow(x, n)

Medium leetcode.com ↗
Idea: sound core, broken execution

Your instinct is exactly the right one: square the base to halve the exponent. That's binary exponentiation, and it's the intended O(log n) solution. The idea is a keeper.

Where it went sideways is the odd-exponent correction. An odd step doesn't add to a counter — it multiplies in the current base once. Tracking it as an integer count and re-injecting x0**count at every level both compounds (positive n) and ignores the sign (negative n), so it lands on the right answer only for certain bit patterns.

That's the whole critique. Keeping the idea, dropping the code — let's rebuild it from the first read.

From scratch — thinking out loud

Pretend I'm at the whiteboard seeing this cold. Here's the actual monologue.

Read itCompute x to the n. Look at the constraints: n can be as large as 2^31 — about two billion — and it can be negative or zero. So the naive "multiply x by itself n times" is O(n): two billion multiplications. Too slow. The real question the problem is asking is: can I compute x^n in far fewer than n multiplications?

Find the leverageI want to reuse work. Here's the identity that unlocks it: x^n = (x·x)^(n/2). One squaring of the base halves the exponent. If I can keep halving, I go from n steps to log₂ n steps — about 31 instead of two billion. That is the entry point. The trigger to memorize: "exponent, and it must be sub-linear" → "halve the exponent by squaring the base."

Handle the odd caseWhat if n is odd? Then n/2 isn't clean. Split off one factor: x^n = x · (x·x)^((n-1)/2). So an odd exponent costs me exactly one extra multiply by the current base, and then it's even again. That single sentence is the entire algorithm — and it's precisely the step the broken version fumbled.

Reframe as binarySquaring-and-halving is just reading n in binary. Position i contributes x^(2^i): the base I get by squaring i times. The answer is the product of x^(2^i) over every position where n has a 1 bit. So I walk the bits from the bottom: keep a running base that I square every step, and whenever the current bit is 1, multiply that base into the answer.

Sign and zeroNegatives: x^(-n) = (1/x)^n. I handle it once, at the top — flip x to 1/x and n to -n — so the core loop only ever sees a non-negative exponent. Zero falls out for free: the loop just doesn't run and the answer stays 1.

Write itA running result = 1, a running base = x. While n isn't zero: if the low bit is set, fold the base into result; then square the base and shift n right. Done.

The algorithm in one sentence On an odd bit, multiply the answer by the current base, once; on every step, square the base and drop a bit. No counter, no sign bookkeeping inside the loop.

The full solution

class Solution:
    def myPow(self, x: float, n: int) -> float:
        if n < 0:              # x^(-n) == (1/x)^n — sign handled ONCE, up front
            x, n = 1 / x, -n
        result = 1.0
        while n:
            if n & 1:          # low bit set -> multiply in the base at this position
                result *= x
            x *= x             # square the base for the next binary position
            n >>= 1            # drop the consumed bit
        return result

Time O(log n) — one iteration per bit of n. Space O(1). Iterative, so no recursion-depth worry at n = 2^31. This exact file ships with a 500-case fuzz suite in test_solution.py.

Takeaway

Pattern · carry forward

When the input is an exponent — or any "apply this operation n times" with n huge — don't iterate n times. Halve the count by combining results: square the base, halve the exponent. The answer is the product of the base at each 1-bit of n.

Same skeleton powers modular exponentiation (pow(a, b, m)), fast matrix power (Fibonacci in O(log n)), and binary lifting (jump 2^k ancestors in a tree). Learn the shape once; it reappears constantly.

And the meta-lesson from the two wrong tries: when a fix makes a wrong solution pass more tests without your being able to state the invariant it now satisfies, that's not progress — it's a better coincidence. Write the one-sentence invariant first; if code doesn't serve it, delete that code.
